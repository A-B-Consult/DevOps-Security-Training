What this training covers

This is engineering-specific security training for our move toward ISO 27001 certification and the rollout of enterprise AI tooling. It focuses on the security decisions you make in your daily work — designing, coding, handling data, using AI, and reporting incidents.

This does not replace general cyber-awareness training (phishing, passwords, social engineering, malware) — you receive that separately.

Key Message
Security is not a final testing activity. It begins during design.
Security is everyone's responsibility
Not a gate owned by one team. Every PR, config and dependency is a security decision.
Secure-by-Design
Safe defaults, least privilege, fail closed. The secure path should be the easy path (ACSC Secure-by-Design).
Security requirements
Write them as acceptance criteria: authZ, validation, logging, data handling — sized to risk.
Threat modelling
Ask "what can go wrong?" early. Who/what are we trusting? What's the worst case if this input is hostile?
Risk-based decisions
Effort scales with impact. An internal dashboard ≠ a payments endpoint. Document the trade-off.
Secure development lifecycle
Security threads through design → code → review → test → deploy → operate, not bolted on at the end.
REAL-WORLD EXAMPLES
CAUSED BY A DESIGN DECISION
An API returns a record by ID with no ownership check. Any user can enumerate IDs and read others' data (IDOR). The flaw is the access-control model, not a typo.
CAUSED BY AN IMPLEMENTATION DECISION
The design required server-side authZ, but one new endpoint shipped without the check copied in. Correct design, incorrect implementation.
In your daily work: before coding a feature, spend two minutes asking "who could misuse this, and what data could leak?" That habit prevents most design-level flaws.

What is personal information?

Any information about an identified individual, or one who is reasonably identifiable — names, emails, IPs, device IDs, location, and combinations that re-identify someone. [Company] is Australian-based and processes personal information of customers, users, employees and partners globally.

The Australian Privacy Principles (APPs), regulated by the OAIC (oaic.gov.au), govern how we collect, use, store and disclose it. Build to Privacy by Design: privacy as a default, not an afterthought.

Data minimisation
Collect and keep only what the purpose needs. Don't store a field "just in case".
Purpose limitation
Use data only for what it was collected for. New use = new basis.
Data retention
Delete when no longer needed. Set TTLs; don't hoard logs and exports forever.
Access controls
Least privilege to personal data. Access should be need-to-know and auditable.
DEVELOPER RESPONSIBILITIES — PRACTICAL
APP LOGSDon't log full request bodies, tokens, or PII. Log identifiers and outcomes, redact the rest.
ANALYTICSVerify what third-party SDKs actually send. Email/IP in event payloads is a disclosure.
PROD SUPPORTAccess production data only with a justified need; never copy records to your laptop or chat.
TEST ENVSUse synthetic or masked data. Real production data in staging is a breach waiting to happen.
3RD-PARTYSharing personal data with a vendor needs an approved basis and contract — check before integrating.
In your daily work: treat personal data as radioactive — minimise it, mask it in non-prod, keep it out of logs, and know where it flows. See oaic.gov.au for the APPs.
Acceptable Use
Use approved AI systems only. Never paste customer data, internal/confidential information, secrets, or proprietary source code into unapproved or public AI tools. When unsure of a tool's classification, ask before you paste.

AI-generated code = same standards as human-written code

Every AI suggestion gets human review, testing, and security review. You are accountable for code you commit. Watch for hallucinated APIs, made-up package names (slopsquatting), insecure defaults, and licence/copyright issues from generated code.

Building AI-enabled features? Secure AI development
• Threat model the AI component explicitly
• RAG security: trust boundaries on retrieved content
• Prompt injection: treat model output as untrusted input
• Data access controls: the model inherits user permissions, not god-mode
• Model security: guard against poisoning, abuse and leakage
• Untrusted outputs: never execute model output without validation
In your daily work: approved tools only, no sensitive data in prompts, and review AI code as if a stranger wrote it. ACSC AI guidance: cyber.gov.au.
Key Message
Early reporting is more important than certainty.
You are not expected to investigate incidents yourself — you are expected to report them promptly. A false alarm costs minutes; a delayed real incident costs far more.
Security incidents
Exposed credentials, unauthorised access, vulnerable production systems.
Privacy incidents
Personal data exposed, sent to the wrong party, or accessed without basis.
AI-related incidents
Sensitive data sent to AI tools, harmful AI output shipped, prompt-injection abuse.
How to report & escalate
  1. Report to ISMS immediately: [email protected]
  2. Tell your Tech Lead / Engineering Manager
  3. For active/serious incidents, escalate without delay
  4. Don't broadcast details in public channels
Preserve evidence
  • Don't delete logs, messages or commits
  • Don't "fix and forget" — note the time you noticed
  • Capture screenshots / IDs
  • Rotate exposed secrets, but tell ISMS too
REPORT THESE — EXAMPLES
An API key or password committed to a repo
Customer data visible to the wrong account
A production system found unpatched / exploitable
Someone pasting customer data into a public AI tool
A suspicious or unexpected third-party package
A privacy data breach (see OAIC data breach response)
In your daily work: if something feels wrong, report it. When in doubt, report — that single habit is the most valuable thing in this module.
